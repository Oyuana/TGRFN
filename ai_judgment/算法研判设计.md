# AI 智能研判 · 算法与设计思想说明

> 面向：答辩 / 技术评审 / 交接。本文讲清楚 **AI 研判这一层「是怎么想的、由哪些模块组成、每个模块为什么这样设计」**，
> 与 TGRFN 主模型（时序图关系假新闻检测，DPSG）配套使用。
>
> 一句话定位：**深度模型负责"算"，规则内核负责"翻译成可解释指标"，大模型(LLM)负责"写成人话报告"，前端负责"讲清楚"。** 三者职责严格分离、互不越权。

---

## 一、为什么需要"AI 研判层"？

TGRFN 主模型（DPSG）本身是一个判别器：输入一条新闻及其传播子图，输出一个假新闻概率。
但在真实的内容安全 / Trust & Safety 场景里，**只给一个概率是不够用的**：

| 现实需求 | 只有概率时的问题 | 研判层的回应 |
|----------|------------------|--------------|
| 审核员要知道"凭什么" | 0.82 这个数字不可解释、不可复核 | 把模型**中间变量**拆成结构冲突、模态贡献、传播规模等**可核查指标** |
| 类别不平衡 | 用 0.5 当阈值会系统性误判 | 用 `odds vs 训练集正负样本比` 做**校准判别** |
| 早期 / 稀疏传播 | 样本太少时模型容易过自信 | 检测稀疏传播并**主动下调置信度** |
| 要出报告 | 人工写报告慢且不一致 | LLM 把指标**自动翻译**成结构化中文报告 |
| 系统要稳 | LLM 会抖动 / 限流 / 无密钥 | **规则兜底 + 自动降级**，接口永远返回结构一致的结果 |

因此 AI 研判层的核心设计原则是：

> **LLM 不重算、不推翻深度模型的任何数值；一切"事实依据"来自模型中间变量 + 规则内核。
> LLM 只把数字翻译成自然语言。** 这样既拿到大模型的表达力，又不引入幻觉风险。

## 二、整体架构与数据流

```text
                         ┌──────────────────────────────────────────────┐
   一条新闻 + 传播子图 → │  Module 1 · 深度模型推理 (model.py)            │
                         │  DPSG.forward(return_internals=True)           │
                         │  额外吐出 ModelInternals：                      │
                         │   probability / dist_mse / rep_scale /         │
                         │   modality_energy / num_neighbors              │
                         └───────────────┬──────────────────────────────┘
                                         ▼
                         ┌──────────────────────────────────────────────┐
                         │  Module 1 · 量化研判内核 (internals.py)        │
                         │  纯函数：中间变量 → JudgmentMetrics            │
                         │   判别间隔 / 假新闻风险 / 结构冲突指数 /        │
                         │   模态注意力 / 稀疏传播 / 预分级 / 置信度       │
                         └───────────────┬──────────────────────────────┘
                                         ▼
                         ┌──────────────────────────────────────────────┐
                         │  Module 2 · LLM 解释层 (prompts.py +           │
                         │            llm_report.py)                      │
                         │  把"只读指标"翻译成结构化中文研判报告           │
                         │  失败自动降级为规则报告(mock)                   │
                         └───────────────┬──────────────────────────────┘
                                         ▼
                         ┌──────────────────────────────────────────────┐
                         │  Module 3 · 接口契约 (api.py)                  │
                         │  POST /api/analyze → { metrics, report }       │
                         └───────────────┬──────────────────────────────┘
                                         ▼
                         ┌──────────────────────────────────────────────┐
                         │  Module 4 · 前端展示 (前端.html)               │
                         │  研判等级 / 拓扑演化 / 指标 / 详细报告 / PDF    │
                         └──────────────────────────────────────────────┘
```

**为什么这样分层？** —— 解耦 + 可替换 + 可复现：
- 每一层只依赖上一层的**数据契约**，不依赖其实现。换模型、换 LLM 厂商、换前端都互不影响。
- 量化内核是**纯函数、零外部依赖（只用标准库 math）**，可以脱离 PyTorch 单独单测（`python -m ai_judgment.internals`）。
- 同一 `news_id` 在 eval 模式下结果可复现。

## 三、模块详解（是什么 + 为什么）

### Module 1-A · 深度模型改造：`model.py`

**做了什么**：给 DPSG 增加一个 `forward(..., return_internals=True)` 旁路。
默认 `return_internals=False` 时**签名完全兼容**原训练脚本 `(predictions, dist)`，训练代码一行不用改；
打开开关后，额外返回 `ModelInternals`：

| 中间变量 | 来源 | 含义 |
|----------|------|------|
| `probability` | `sigmoid` 输出 | 越接近 1 越"真" |
| `dist_mse` | `MSELoss(align_c, align_g)`，即训练里的 `loss_dis` | 内容表示↔传播图结构表示的对齐误差 |
| `rep_scale` | `0.5·(‖align_c‖²+‖align_g‖²)` 均值 | 表示能量，用于把 MSE 归一化成尺度无关 |
| `modality_energy` | 前向时缓存的 text/image/time 三模态表示能量 | 各模态对判别的贡献量 |
| `num_neighbors` | 参与聚合的邻居规模 | 传播规模，用于稀疏判定 |

**为什么这样设计**：
- **不动训练、不动权重**：研判能力以"旁路 hook"形式挂上去，对已训练好的 checkpoint 零侵入。
- **复用模型已有的物理量**：`align_c`（内容对齐表示）与 `align_g`（传播图对齐表示）本就是 DPSG 为对齐损失而算的，
  我们直接拿它们的差异当"结构冲突"信号，**不引入任何新的、未经训练的量**——这是可信度的关键。

### Module 1-B · 量化研判内核：`internals.py`

把上面 5 个原始中间量，经一组**有明确物理意义的规则**，汇聚成 `JudgmentMetrics`。逐项说明：

**1) 判别间隔与假新闻风险（为什么不用 0.5）**

```
margin = logit(p) − log(ratio)          # ratio = 训练集 真新闻数/假新闻数
predicted_label = REAL if margin ≥ 0 else FAKE
fake_probability = sigmoid(−margin · scale)   # margin 越负，风险越高
```

- 真实判别规则是 `odds = p/(1−p)` 与 `ratio` 比较（等价于 `logit(p)` 与 `log(ratio)` 比较），
  **这正是 GossipCop 训练脚本里用的判别方式**。
- **为什么**：类别不平衡时，"0.5 阈值"会系统性偏向多数类；用 `ratio` 做基准等价于**把先验比例校准掉**，
  让"偏离基准的程度"而不是"绝对概率"来决定风险。`margin` 离 0 越远，结论越稳健。

**2) 结构冲突指数（水军 / 异常传播图谱）**

```
structural_conflict_index = dist_mse / (rep_scale + ε)
alert = index ≥ 阈值(默认0.5)
```

- **物理含义**：当一条新闻的**内容语义表示**和它**实际传播图结构表示**严重不一致时，
  说明"内容很普通，但它被转发/互动的方式很不自然"——这正是**水军刷量、协同操纵、异常传播图谱**的典型特征。
- **为什么用 `dist_mse / rep_scale`**：原始 MSE 受表示尺度影响，除以平均表示能量后得到**尺度无关的相对冲突度**，
  阈值才能跨样本通用。

**3) 模态注意力贡献**

```
modality_attention = softmax(modality_energy)   # {text,image,time} 求和=1
dominant = argmax;  alert = max ≥ 阈值(默认0.6)
```

- **为什么 softmax**：把三模态能量归一化成"占比"，可直接喂饼图/雷达图，也便于设统一阈值。
- **可解释告警**：
  - `image` 主导且超阈值 → 提示**视觉特征篡改嫌疑**（配图造假 / 移花接木）。
  - `time` 主导且超阈值 → 提示**爆发式时间衰减**（典型谣言扩散节律）。
  - `text` 主导 → 判别主要由文本驱动，建议核查文本事实。

**4) 传播稀疏 → 置信度修正**

```
sparse = num_neighbors < 阈值(默认10)
confidence = min(1, |margin|/3 + 0.3);  if sparse: confidence ×= 0.7
```

- **为什么**：早期 / 稀疏传播时样本太少，模型容易过自信。检测到稀疏就**主动打 7 折**并提示"持续监测"，
  对应 TGRFN 关注的**早期谣言识别**场景——宁可保守，不给过绝对的结论。

**5) 量化预分级**

```
fake_prob < 0.35           → SAFE
0.35 ≤ fake_prob < 0.65    → SUSPICIOUS
fake_prob ≥ 0.65           → HIGH_RISK
结构冲突告警时整体上调一档
```

- **为什么保留一个"规则预分级"**：它是不依赖 LLM 的**保守基线/兜底**。即便 LLM 不可用，
  系统也能给出确定的风险档位；同时它作为 LLM 的"重要参考"，约束 LLM 不要乱跳级。

> **设计取舍**：所有阈值集中在 `config.py`（`RISK_TIERS`、`STRUCTURAL_CONFLICT_ALERT`、
> `MODALITY_DOMINANCE_ALERT`、`SPARSE_PROPAGATION_THRESHOLD`、`DATASET_RATIO`…），
> 不在业务代码里写死，部署不同数据集时只调配置。

### Module 1-C · 推理引擎与数据集成：`inference.py` / `data_adapter.py`

- **进程级单例**加载权重（`InferenceEngine.instance()`），避免每个请求重复 `load checkpoint`。
- `analyze(news_id)`：组装 `(node, neighbor_order, all_t)` → 跑一次 `forward(return_internals=True)` →
  调 `compute_judgment_metrics`，把模型中间量与 `config` 阈值一起喂进纯函数内核。
- **唯一集成点 `load_data_bundle()`**：把训练机的落盘格式（复用 GossipCop.py 的
  `neighbor_loader/data_loader`）装配成 `DataBundle`。数据没挂载时返回 `None`，
  接口以 **503 明确报错**而不是静默出错。

**为什么**：把"与训练机数据格式强绑定"的部分**收敛到一个函数**，其余链路全部与数据无关，
交接 / 换数据集时只动这一个点。

### Module 2 · LLM 解释层：`prompts.py` + `llm_report.py`

**职责**：把"只读的量化指标"翻译成结构化中文研判报告，字段固定：
`Risk_Level / Conclusion / Evidence[] / Propagation_Analysis / Recommendation`。

**三条铁律（写进 System Prompt）**：
1. **不得重算、不得推翻**材料里的任何数值（概率、结构冲突、模态贡献都是模型事实）。
2. **不得编造**材料里没有的证据/来源/数据，证据只能引用 `triggered_rules` 和给定指标。
3. **严格输出 JSON**，便于前端直接消费。

**为什么这样约束 LLM**：这是**反幻觉**的核心。LLM 强在"组织语言"，弱在"算数/守事实"；
所以我们只让它做擅长的事——把已经算好、可核查的数字写成专业、分层、可读的报告。

**稳健性设计（工程关键）**：
- **provider 抽象**：`anthropic` / `openai` / `mock` 三种实现同一函数签名，配置切换。
- **自动降级**：真实调用一旦失败（网络/限流/无密钥/解析异常），**自动回退 `mock`**，
  并在报告里标注 `_provider` / `_error`。**接口永远返回结构一致的 report**，前端不会因 LLM 抖动而崩。
- **mock 不是占位符**：规则版 `_mock_report` 也会**逐项引用具体数值**生成详细报告
  （判别间隔、结构冲突、模态占比、传播规模 → 证据/传播分析/分层处置建议），
  保证离线 / 无密钥也能拿到内容充实的报告。
- **JSON 兜底解析**：容忍 ```json 代码块包裹或前后多余文本，`_extract_json` 稳健抽取。

### Module 3 · 接口契约：`api.py` / `API_CONTRACT.md`

`POST /api/analyze` 返回**两段**：

```jsonc
{
  "metrics": { ...底层模型量化指标... },   // 可直接喂仪表盘 / 雷达图 / 警示条
  "report":  { ...LLM 文本研判报告... }      // 给人看的结论 / 证据 / 建议
}
```

**为什么 metrics 与 report 分离**：
- `metrics` 是**机器可读、可校验**的硬数据（前端画图、阈值判断都靠它）；
- `report` 是**人可读**的软解释。两者来源不同（一个来自模型+规则，一个来自 LLM），
  分开返回让前端能在 LLM 不可用时**仅凭 metrics 也能完整展示**。
- 提供 `with_report=false` 快速模式（只算 metrics，不调 LLM），用于联调/压测。

### Module 4 · 前端展示：`前端.html`

- **研判界面**：左侧原始新闻信息 + 运行轨迹日志；中间 **t→t+60 传播拓扑动态演化**（可拖时间轴回看）；
  右侧四个标签页（判定 / 指标 / 管理 / 报告）。
- **研判等级条**：用 `report.Risk_Level`（高危/存疑/安全）着色，头部两要点直接显示
  **结构冲突指数**与**主导模态**（取自 metrics），自适应换行、完整显示。
- **报告页**：风险徽标 + 核心结论 + 关键证据列表 + 传播/结构分析 + 处置建议 + 量化指标快照 + 模态贡献条，
  并可一键导出 PDF。
- **零依赖演示**：内置 `window.DEMO`（3 个精选案例 + 可复现的传播图生成器 + 详细报告），
  `USE_LIVE_API=false` 时纯前端即可展示最佳效果；接后端时优先后端、不可用自动回退，演示不白屏。

## 四、与 TGRFN 主模型思想的衔接

AI 研判层并非独立发明指标，而是**把 TGRFN 已有的建模思想"显式化、可解释化"**：

| TGRFN 建模思想 | 在研判层的对应体现 |
|----------------|--------------------|
| 多关系（news/post/user）传播图建模 | 结构冲突指数（内容表示 ↔ 传播图结构表示的对齐误差） |
| 传播节点的动态贡献衰减 / 时序建模 | 时间模态贡献 + 爆发式时间衰减告警 |
| 多模态（文本/视觉/时间）特征 | 模态注意力归一化贡献 + 单模态异常主导告警 |
| 早期、稀疏传播下的检测 | 稀疏传播检测 → 置信度下调 + 持续监测建议 |
| 类别不平衡的判别 | `odds vs ratio` 校准判别（而非 0.5 阈值） |

## 五、设计取舍小结（为什么这么做）

1. **可解释优先**：所有结论都能回溯到一个具体数值/告警，审核员可复核——这是内容安全场景的底线。
2. **LLM 只做翻译、不做裁决**：用三条铁律 + 只读输入隔绝幻觉，把可信度交给深度模型与规则。
3. **稳健降级**：mock 兜底 + provider 抽象 + 503 明确报错，让系统在 LLM / 数据缺位时仍可用且诚实。
4. **解耦与可复现**：纯函数内核、数据契约分层、单一集成点、可复现推理，便于单测、交接和替换。
5. **配置驱动**：阈值/密钥/路径全部集中在 `config.py`，换数据集 / 换厂商不改业务代码。
